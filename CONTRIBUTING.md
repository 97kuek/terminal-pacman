# コントリビューションガイド

このプロジェクトへの変更は、遊べる状態を壊さずに少しずつ改善する方針で進めます。

## 開発環境

### Windows

MinGW-w64 GCC を使います。

```powershell
powershell -ExecutionPolicy Bypass -File .\build.ps1
```

実行:

```powershell
.\build\terminal-pacman.exe
```

### Linux / macOS

```sh
make
./build/terminal-pacman
```

## 変更前に読むもの

- `SPEC.md`: 現在のゲーム仕様
- `CLAUDE.md`: 全体ガイド（ビルド・構成・調整ポイント・注意点）
- `docs/IMPLEMENTATION_PLAN.md`: 実装構成と進捗
- `docs/ROADMAP.md`: ゲーム性改善の優先度
- `docs/AI_DESIGN.md`: 敵 AI の設計
- `AGENTS.md`: AI エージェント向けの作業方針

## コーディング方針

- C99 を基準にし、`-Wall -Wextra -pedantic` で警告ゼロを保つ。
- 外部ライブラリは、必要性が明確になるまで追加しない。
- OS 依存処理（入力・VT初期化・描画出力・端末サイズ・サウンド・スリープ）は `src/platform.h` の API に閉じ込める。
- ゲームルールは `src/game.c` に寄せ、描画処理と混ぜない。
- 敵の経路探索は `src/pathfind.c`（A*）に分離する。
- 描画は `src/render.c` に寄せる。表示は ASCII グリフ + ANSI カラーを使う。
- 迷路を変えるときは `levels/*.txt` と `src/game.c` の内蔵配列を `tools/gen_levels.py` で生成して同期させる（連結性はジェネレータが検証）。
- 関数は短く保ち、状態変更の責務を分ける。

## 変更の進め方

1. 変更したい内容を `SPEC.md` または `docs/ROADMAP.md` と照らし合わせる。
2. 仕様が変わる場合は、実装前にドキュメントを更新する。
3. 実装する。
4. ビルドする。
5. テストを実行する（`make test` または `powershell -File build_tests.ps1`）。ロジックを変えたらテストも追加・更新する。
6. 可能なら実際にプレイして、主要な操作を確認する。
7. 変更内容に合わせて README や計画書を更新する。

## 確認項目

最低限、以下を確認してください。

- Windows でビルドできる。
- 変更が平台固有でない場合、Linux / macOS でもビルドできる設計になっている。
- `WASD` 操作が壊れていない。
- `Q` で終了できる。
- 端末モードが通常終了時に復元される。
- プレイヤーと敵が壁を通り抜けない。

## コミットメッセージ

短く、何を変えたかが分かる英語または日本語にします。

### 基本ルール

- 1 行目は 50 文字程度を目安にする。
- 1 行目は「何をしたか」が分かる命令形または要約形にする。
- 句点は付けなくてよい。
- 1 コミットには、できるだけ 1 つの目的だけを入れる。
- 実装とドキュメント更新が同じ仕様変更に属する場合は、同じコミットに含めてよい。
- 無関係な整形、リネーム、機能追加を 1 つのコミットに混ぜない。
- 英語と日本語のどちらでもよいが、同じコミット内では混ぜすぎない。

### 推奨フォーマット

通常は次のどちらかを使います。

```text
<動詞> <対象>
<対象>を<変更内容>
```

英語の動詞例:

- `Add`
- `Fix`
- `Update`
- `Refactor`
- `Document`
- `Remove`

日本語の例:

- `ハイスコア保存を追加`
- `敵AIの移動判定を修正`
- `READMEのビルド手順を更新`

### 本文を書く場合

1 行目だけで理由や注意点が伝わらない場合は、空行を 1 つ挟んで本文を書く。

本文には次を書く。

- なぜ変更したか
- 仕様上の注意点
- 既知の制約
- 手動確認した内容

### 良い例

```text
Add power pellet mode

パワーエサ取得中は敵を食べられるようにする。
終了前は敵表示を点滅させる。
```

```text
敵の移動ロジックを分離

A*を使う追跡型と、進行方向を読む先読み型を分ける。
```

### 避けたい例

```text
update
fix
いろいろ修正
WIP
最終版
```

## 優先して歓迎する変更

- ゲーム性が明確に上がる小さな機能
- 純粋なゲームロジックのテスト追加
- Linux / macOS でのビルド確認
- 端末表示のちらつき軽減
- README や仕様書の改善

## 避けたい変更

- ゲーム性と無関係な大規模リファクタリング
- クロスプラットフォーム性を壊す変更
- 仕様書と実装の差分を放置する変更
- 初期方針と矛盾する重い依存ライブラリの追加
