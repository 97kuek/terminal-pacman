# 敵 AI 設計

`terminal-pacman` の敵AIは2つのアルゴリズムで設計している

1. **A\* 探索による追跡**（全モード共通）(`src/pathfind.c`)
2. **オンライン Q 学習による適応的な追跡**（Endless モードの 1 体のみ(`src/qghost.c` / `src/qfeatures.c`)

- 座標系はグリッドで、各セルは**壁か通行可のどちらか**
- 移動は上下左右の4方向、1手1セル、コストは一律 1
- 左右端には**ワープトンネル**があり水平方向に巻き戻る

---

## 1. A\* 探索（`src/pathfind.c`）

### 何をするか

- ゴーストの現在地 `start` から目標セル `goal` までの**最短経路の最初の1歩**を返す

```c
int astar_next(const char *map, int stride, int width, int height,
               PathPoint start, PathPoint goal, PathPoint *next);
```

### アルゴリズム

A\* は「実コスト `g` ＋ ゴールまでの推定コスト `h`」が小さいノードから展開する探索

- `g(n)`: `start` から `n` までの実コスト
- `h(n)`: `n` から `goal` までの推定コスト
  - 本実装は**マンハッタン距離** `|dx| + |dy|`でヒューリスティック関数を実装
  - 4方向・一律コストの格子では実距離を超えないため、A\* は**最短経路を保証**
- `f(n) = g(n) + h(n)`: 優先度。`f` が最小のノードを次に展開する。


### 実装メモ（`pathfind.c`）

- グリッド最大 64×32。`g_score` / `came_from` / `closed` / `open_set` は固定長配列（動的確保なし）。
- open 集合は線形走査で最小 `f` を選ぶ（このサイズでは十分軽い）。
- 経路復元は `came_from` を `goal` から逆にたどり、**`start` の次のセル**だけを `*next` に返す（ゴーストは 1 手ずつ動くため）。
- `start == goal` は 1、`goal` が壁・到達不能は 0 を返す。

### なぜ BFS ではなく A\*

- このゲームは 1 手一律コストなので、BFSでも A\* でも**得られる最短経路は同じ**
- 当初は BFS を使用していたが、迷路を密にして敵を4体に増やし、**各敵が毎ステップ自前の目標へ経路探索する**ようになって探索頻度が大きく増えたため、ゴール方向に展開を絞れる A\* に切り替えた

### 逃走（パワーエサ中）

- パワーエサ中の frightened 状態では「目標へ近づく」ではなく「プレイヤーから**最も遠ざかる**」必要がある
- これは A\* ではなく、プレイヤー始点の **BFS 距離マップ**（`build_distance_map`）を作り、隣接セルのうち距離が最大の方向を選ぶ（`choose_frightened_direction`）。

## 2. ゴーストの性格と全体制御（`src/game.c`）

- 通常追跡時、4体はそれぞれ異なる**目標セル**を A\* に渡す

| 敵 | 方針 | 追跡時の目標 |
| --- | --- | --- |
| CHASER（Blinky） | 直線追跡 | プレイヤーの現在位置 |
| AMBUSH（Pinky） | 待ち伏せ | プレイヤーの進行方向数マス先 |
| FLANK（Inky） | 挟み込み | `2 × (プレイヤーの少し先) − CHASER の位置`（CHASER を支点に外挿） |
| SHY（Clyde） | 臆病 | 遠いときは現在位置、近づくと自分のコーナーへ退く |

- 目標が壁・場外になる場合（FLANK の外挿など）は、`nearest_open_cell` で最も近い通行可セルへスナップしてから A\* にかける

### 散開 / 追跡ウェーブ

- 緊張の山場を作るため、全敵が周期的にモードを切り替える

- **SCATTER**: 各敵が自分のホームコーナーへ向かい盤面に散る
- **CHASE**: 各敵が上記の性格に従ってプレイヤーへ収束する

切り替え間隔は難易度別の `chase_ticks` / `scatter_ticks`。パワーエサ中はウェーブを凍結し、全敵が frightened（逃走）になります。プレイヤーが**スタシス・パルス**を発動中（`stasis_ticks > 0`）は、全敵の移動処理をスキップして固定します。

### 速度

各敵は自分のテンポ（`ghost_speed`）で動き、群れが波状に到達します。

- 基本テンポ `game_ghost_move_interval`（収集が進むほど短く＝速く）。
- 難易度（Easy/Normal/Hard）で `interval_delta` を加算。
- frightened 中は 1 段遅く。
- **CHASER** は残ドットが 1/4 未満で 1 段速く（Cruise Elroy）。
- **Endless** では `mazes_cleared` に応じてさらに加速。

## 3. オンライン Q 学習ゴースト（`src/qghost.c` / `src/qfeatures.c`）

Endless モードでは **ghost 0** が A\* の代わりに**表型 Q 学習**で動き、プレイヤーの動きの癖に 1 プレイ内で適応します（盤面ではマゼンタ表示）。Classic / Time Attack は決定的 AI のまま。

### 強化学習としての定式化

- **状態 `s`**: ゴーストから見たプレイヤーの相対位置（粗いバケット）× プレイヤーの進行方向。`qfeat_state`。
  - `dx = px − gx`, `dy = py − gy` をそれぞれ 5 段にバケット化（`d<-3, -3..-1, 0, 1..3, >3`）。
  - プレイヤーの向き（停止/上/下/左/右 = 0..4）を加える。
  - `state = (dxb × 5 + dyb) × 5 + player_dir` → **125 状態**（テーブルは余裕を見て 256 確保）。
  - プレイヤーの向きを状態に含めることで「この方向に動く癖」を学習対象にできます。
- **行動 `a`**: 上/下/左/右 の 4 つ。壁・場外に出る行動は合法マスク `valid[a]` で除外（`qfeat_valid`、水平ラップ対応）。
- **報酬 `r`**: `qfeat_reward`。プレイヤーへのマンハッタン距離が**縮んだ分**（`old_d − new_d`）＋ 重なったとき `+5`（捕獲ボーナス）。
- **遷移**: 行動後に次状態 `s'` を再計算。

### ε-greedy 行動選択（`qghost_select`）

```text
ε = 0.05 + 0.45 / (1 + steps / 400)        // 試行が進むほど 0.5 → 0.05 に減衰
確率 ε で:   合法な行動から一様ランダムに選ぶ（探索）
それ以外:    合法な行動のうち Q[s][a] が最大のものを選ぶ（活用）
```

序盤は探索的（ゆるい）、学習が進むと活用的（鋭い）になります。

### Q 学習の更新（`qghost_update`）

標準的な Q 学習（時間差分・off-policy）の 1 ステップ更新:

```text
Q[s][a] ← Q[s][a] + α · ( r + γ · max_{a' が合法} Q[s'][a']  −  Q[s][a] )
```

- 学習率 `α = 0.2`、割引率 `γ = 0.9`。
- 次状態に合法行動が無ければ将来項は 0。
- 状態・行動が範囲外なら何もしない（安全）。

### ゲームへの統合（`game.c`・`move_learner_ghost`）

毎ステップ「観測 → 選択 → 移動 → 報酬計算 → 更新」を 1 回行います。

```text
s   = qfeat_state(ghost, player, player_dir)
valid = qfeat_valid(map, ghost)
a   = qghost_select(Q, s, valid)
ghost を a の方向へ移動（ラップ対応）
r   = qfeat_reward(old_ghost, new_ghost, player)
s'  = qfeat_state(ghost', player, player_dir)
qghost_update(Q, s, a, r, s', qfeat_valid(map, ghost'))
```

状態符号化・合法判定・報酬は `src/qfeatures.c` に切り出し、**ゲーム内学習器とオフライン学習ツールが同じ定義**を使います。

### オフライン事前学習とウォームスタート

- `tools/qsim.c`（`make qsim` で実行）: 盤面（`maze_generate`）＋スクリプト化した「癖を持つプレイヤー」に対して 1 体の学習器を多数エピソード走らせ、学習済み Q テーブルを `src/qtable_data.h` に書き出します。
- ゲームは Endless 開始時にこのテーブルで**ウォームスタート**し（`memcpy`）、初手から有能に追跡してから online で適応します（毎回ゼロ初期化のランダム探索を避ける）。

### 評価結果（正直な所）

`qsim` は学習が効いているかを計測します。現状の単体追跡タスクは**ほぼ天井**で、最初から捕獲率が高く、学習による伸びはほとんど見られません。

| 区間 | 平均ステップ（捕獲まで） | 捕獲率 |
| --- | --- | --- |
| 最初の 400 エピソード | 54.1 | 91.5% |
| 最後の 400 エピソード | 56.6 | 90.5% |

つまり「1 体で小さな迷路のプレイヤーを追う」だけなら、報酬整形のおかげで**学習しなくても十分強い**ため、Q 学習の上積みは小さい、という結果です。ウォームスタートの価値は「天井を上げる」ことではなく「序盤のランダム探索を無くす」ことにあります。

### 今後の改善余地

- A\* が追跡を最適化済みなので、Q 学習の価値が出るのは「単純追跡」ではなく**プレイヤーの癖の先読み・複数体の連携・動的難易度調整**。
- 状態をより豊かに（直近の移動履歴、分岐構造、他ゴーストの位置）、報酬を癖の予測に寄せる、評価をコールド vs ウォームの比較にする、などで「学習が効く региーム」を作れます。
